# GCR 동시 파일 업로드 처리 능력 분석

## 현재 배포 설정

```bash
--cpu 1 --memory 2Gi
--max-instances 3
--concurrency (기본값 80)
--timeout 900
```

### 애플리케이션 설정
- **번역 워커 수**: 5개 (ThreadPoolExecutor)
- **API 타임아웃**: 30초
- **최대 파일 크기**: 16MB
- **세션별 락**: 중복 요청 방지 (같은 세션만)

---

## 파일 업로드 처리 단계별 소요 시간

### 일반적인 NOTAM PDF (100-200개 NOTAM 포함) 기준

| 처리 단계 | 소요 시간 | 리소스 사용 | 설명 |
|----------|----------|------------|------|
| **1. 파일 저장** | 0.1-0.5초 | 낮음 | 파일 I/O |
| **2. PDF 텍스트 변환** | 1-3초 | CPU 중간 | pdfminer 사용 |
| **3. NOTAM 필터링** | 1-2초 | CPU 낮음 | 텍스트 파싱 |
| **4. AI 번역 (Gemini)** | **10-60초** | **CPU 높음** | **가장 큰 병목** |
|   - 워커 5개 병렬 처리 | - | - | 각 NOTAM마다 2개 API 호출 (영어+한국어) |
|   - NOTAM당 평균 | 0.5-1초 | - | API 응답 시간 포함 |
| **5. 시간 변환/포맷팅** | 0.5-1초 | 낮음 | 시간대 변환 |
| **총 처리 시간** | **13-67초** | - | NOTAM 수에 따라 변동 |

### 리소스 집약도가 높은 이유
1. **Gemini API 호출**: 외부 API 호출 (네트워크 I/O + 대기)
2. **병렬 처리**: 워커 5개 × 각 NOTAM 2개 API = CPU/메모리 사용
3. **PDF 변환**: CPU 집약적 작업

## 동시 파일 업로드 처리 가능 인원

### 이론적 최대 동시 요청

```
최대 인스턴스 수 × 인스턴스당 동시성 = 3 × 80 = 240개 동시 요청
```

### 실제 파일 업로드 처리 능력

#### 시나리오 1: 짧은 처리 시간 (평균 20초)
- 각 요청이 20초 동안 리소스를 점유
- 1개 인스턴스가 80개 요청 처리 중이라면:
  - 실제로는 시간당 약: 80 × (60초/20초) = **240개 요청 처리 가능**
- 3개 인스턴스: **약 720개 요청/시간** (초당 0.2개)

#### 시나리오 2: 긴 처리 시간 (평균 60초)
- 각 요청이 60초 동안 리소스를 점유
- 1개 인스턴스: 80 × (60초/60초) = **80개 요청/시간**
- 3개 인스턴스: **약 240개 요청/시간** (초당 0.067개)

### 동시 파일 업로드 가능 인원 (실질적)

**주요 제약 요인:**
1. **CPU 리소스 경쟁**: 1 코어로 80개 동시 요청 처리 시 CPU 부족
2. **메모리 사용**: 각 요청이 약 50-100MB 사용 시 최대 약 20-40개만 처리 가능
3. **번역 워커 경쟁**: 인스턴스당 5개 워커 공유

**현실적인 동시 처리 능력:**

| NOTAM 수 | 평균 처리 시간 | 동시 업로드 가능 | 초당 처리량 |
|---------|--------------|----------------|------------|
| 50개 이하 | 15-25초 | **10-15명** | 0.4-1.0개/초 |
| 100-150개 | 30-50초 | **5-8명** | 0.1-0.27개/초 |
| 200개 이상 | 60-90초 | **3-5명** | 0.033-0.083개/초 |

**추정:** 현재 설정으로는 **약 5-10명**이 동시에 파일 업로드하여 처리 가능

## 10명 동시 접속 시 파일 업로드 처리 분석

### 시나리오: 10명이 동시에 파일 업로드

#### 케이스 1: 모두 짧은 파일 (50개 이하 NOTAM, 20초 처리)

**처리 흐름:**
```
시간 | 인스턴스 1 (80 요청 슬롯) | 인스턴스 2 (80 요청 슬롯) | 인스턴스 3 (80 요청 슬롯)
-----|-------------------------|-------------------------|-------------------------
0초  | 10개 요청 도착           | -                       | -
0-20초| 10개 요청 처리 중 (CPU 100%) | -                    | -
20초 | 10개 완료 ✅             | -                       | -
```

**결과:**
- ✅ **문제 없음**: 10개 요청이 한 인스턴스에서 처리 가능
- 평균 응답 시간: 20초 (정상)
- 속도 저하: **없음** 또는 **미미함** (5% 이하)

#### 케이스 2: 모두 중간 파일 (100-150개 NOTAM, 40초 처리)

**처리 흐름:**
```
시간 | 인스턴스 1 | 인스턴스 2 | 인스턴스 3
-----|-----------|-----------|-----------
0초  | 10개 요청  | -         | -
0-40초| 10개 처리 중 (CPU 100%, 메모리 80%) | - | -
```

**결과:**
- ⚠️ **CPU 부하 증가**: 1 코어로 10개 요청 동시 처리 시 경쟁 발생
- 평균 응답 시간: **40-50초** (약 10-25% 증가)
- 속도 저하: **중간** (10-25%)

#### 케이스 3: 모두 긴 파일 (200개 이상 NOTAM, 60초 처리)

**처리 흐름:**
```
시간 | 인스턴스 1 | 인스턴스 2 | 인스턴스 3
-----|-----------|-----------|-----------
0초  | 10개 요청  | -         | -
0-60초| 10개 처리 중 (CPU 과부하, 메모리 90%) | - | -
```

**결과:**
- ❌ **심각한 속도 저하**: 
  - CPU 경쟁으로 각 요청 처리 시간 증가
  - 번역 워커 5개를 10개 요청이 공유 → 대기 시간 증가
  - 메모리 부족 가능성 (2Gi ÷ 10 = 약 200MB/요청)
- 평균 응답 시간: **80-100초** (약 33-67% 증가)
- 속도 저하: **심각** (30-70%)

### 속도 저하 원인 분석

#### 1. CPU 경쟁
- **현재**: CPU 1 코어
- **문제**: 10개 요청이 동시에 CPU 사용
- **영향**: 각 요청의 처리 시간 20-40% 증가

#### 2. 번역 워커 경쟁
- **현재**: 인스턴스당 워커 5개
- **문제**: 10개 요청이 5개 워커를 공유
- **영향**: NOTAM 번역 대기 시간 증가 (약 2배)

#### 3. 메모리 압박
- **현재**: 메모리 2Gi
- **문제**: 각 요청 약 100-200MB 사용 시 10개 = 1-2Gi
- **영향**: 메모리 부족 시 스왑 발생 → 성능 저하

#### 4. 외부 API (Gemini) 제한
- **문제**: 같은 API 키로 동시 요청이 많으면 제한 가능
- **영향**: API 호출 지연 또는 실패

## 예상 성능 저하 정리

### 10명 동시 파일 업로드 시

| NOTAM 수 | 정상 처리 시간 | 동시 처리 시 예상 시간 | 속도 저하 | 평가 |
|---------|--------------|---------------------|----------|------|
| 50개 이하 | 20초 | 21-22초 | **5-10%** | ✅ 양호 |
| 100-150개 | 40초 | 45-50초 | **12-25%** | ⚠️ 보통 |
| 200개 이상 | 60초 | 80-100초 | **33-67%** | ❌ 심각 |

### 인스턴스 스케일 아웃 영향

**자동 스케일 아웃 시나리오:**
- 트래픽 급증 시 새 인스턴스 생성까지 **10-30초** 소요
- 그 사이 들어온 요청들은 기존 인스턴스가 처리
- 초기 10개 요청 중 일부는 새 인스턴스에서 처리 가능

**개선 효과:**
- 10명 요청이 2-3개 인스턴스로 분산
- 속도 저하: **5-15% 감소** (케이스 2 기준)

## 권장 사항

### 현재 설정에서 10명 동시 사용 시

#### 1. 예상 성능
- **짧은 파일 (50개 이하 NOTAM)**: ✅ **거의 영향 없음** (5-10% 저하)
- **중간 파일 (100-150개)**: ⚠️ **보통 속도 저하** (20-30% 저하)
- **긴 파일 (200개 이상)**: ❌ **심각한 속도 저하** (50-70% 저하)

#### 2. 개선 방안

**즉시 적용 가능 (비용 증가 없음):**
```bash
# 최소 인스턴스 설정 (Cold Start 방지)
--min-instances 1
```

**성능 개선 (비용 증가):**
```bash
# CPU 증가
--cpu 2 --memory 2Gi

# 또는 전체 리소스 증가
--cpu 2 --memory 4Gi
--max-instances 5  # 더 많은 인스턴스
```

**코드 최적화:**
- 번역 워커 수 조정 (환경변수로 제어)
- 캐시 활용도 향상
- 배치 처리 최적화

## 결론

### 현재 설정 기준

**동시 파일 업로드 가능 인원:**
- **최대**: 약 **5-10명** (파일 크기에 따라 변동)
- **10명 동시 접속 시**: 
  - 짧은 파일: ✅ **거의 문제 없음**
  - 긴 파일: ❌ **심각한 속도 저하 (50-70%)**

### 실무 권장사항

1. **현재 설정 유지**: 
   - 동시 사용자 5명 이하
   - 주로 짧은 파일 처리

2. **리소스 증가 필요**: 
   - 동시 사용자 10명 이상 예상
   - 긴 파일 처리 빈번
   - `--cpu 2 --memory 2Gi` 권장 (비용 1.7배)

3. **모니터링 필요**:
   - Cloud Run 메트릭으로 실제 처리 시간 확인
   - CPU/메모리 사용률 모니터링
   - 인스턴스 스케일링 패턴 분석
